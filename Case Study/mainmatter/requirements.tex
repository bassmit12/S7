% ============================================================
\chapter{Requirements Analysis}
\label{chapter:requirements}
% ============================================================

This chapter applies TOGAF ADM Phase B (Business Architecture) to derive requirements from the enterprise context, stakeholder concerns and identified risks defined in Chapter~\ref{chapter:enterprise-context}. Requirements are not generated from technical preferences; each requirement is explicitly traced to a source.

% -------------------------------------------------------
\section{Management Requirements}
% -------------------------------------------------------

\label{tab:req-mgmt}Management requirements address how the IT team and senior leadership observe and control the infrastructure. The IT Director requires a single monitoring dashboard aggregating status, performance, and alerts from both sites (\textbf{MR-01}), with automated alerting notifying on-call IT staff within two minutes of a critical component failure (\textbf{MR-02}). To serve non-technical stakeholders, the Operations Director requires read-only, role-restricted operational dashboards for the Order System and Warehouse that can be used without IT involvement (\textbf{MR-03}). The CEO and Board require automatically generated monthly availability reports for the Customer Order System and customer portal to support SLA validation (\textbf{MR-04}). All administrative access to production servers must be logged and traceable to an individual IAM identity (\textbf{MR-05}), and infrastructure capacity data --- CPU, memory, and storage trends --- must be retained for a minimum of 13 months to support annual capacity planning cycles (\textbf{MR-06}).

% -------------------------------------------------------
\section{User Requirements}
% -------------------------------------------------------

\label{tab:req-user}User requirements express what different groups of people need the infrastructure to do for them. The Customer Service team requires the order portal to be accessible 24/7, with a status page publishing degraded-mode notifications when planned maintenance is in progress (\textbf{UR-01}). Distribution staff at both sites require the Warehouse Management System to remain operational during a WAN outage, necessitating an offline mode at Groningen (\textbf{UR-02}). The Finance Manager requires a complete, immutable audit trail of all financial transactions accessible to authorised Finance staff for at least seven years, as required by Dutch fiscal law (\textbf{UR-03}). The HR Manager requires role-restricted access to the HR/Payroll system such that no IT staff member can access payroll data without explicit HR Manager authorisation (\textbf{UR-04}). IT staff require a jump host with session recording as the sole path to production servers, enabling post-incident forensic review of all administrative sessions (\textbf{UR-05}). Finally, retail customers require that the portal respond within two seconds for standard order operations under normal load conditions (\textbf{UR-06}).

% -------------------------------------------------------
\section{Functional Requirements}
% -------------------------------------------------------

\label{tab:req-functional}Functional requirements define specific system behaviours that the infrastructure must support. The Order System must handle at least 10,000 customer transactions per day during seasonal peaks without service degradation (\textbf{FR-01}). Inventory events from both sites must be synchronised via the RabbitMQ message broker, with guaranteed delivery and a replay capability on WAN recovery (\textbf{FR-02}). Invoice creation events must be published asynchronously by the Order System so that accounting processing does not block the order submission path (\textbf{FR-03}). Payroll data must be encrypted at rest and in transit, with all access logged at the application level including user identity, timestamp, and action (\textbf{FR-04}). All application access must require SSO authentication through the IAM, and direct database or server access that bypasses the IAM is not permitted (\textbf{FR-05}). Metrics, logs, and health data from all infrastructure components must be collected centrally in the Management VLAN (\textbf{FR-06}). Finally, the Groningen WMS node must queue write operations locally and replay them to the central cluster when the WAN link is restored (\textbf{FR-07}).

\section{Non-Functional Requirements}
% -------------------------------------------------------

\subsection{Availability}

\label{tab:req-availability}Availability targets are set per system, reflecting the criticality analysis from Chapter~\ref{chapter:enterprise-context}. The Customer Portal (\textbf{NFR-AV-01}) and Order System (\textbf{NFR-AV-02}) both carry a 99.9\% target, permitting no more than 8.7 hours of downtime per year; this reflects their status as revenue-critical, customer-facing systems. The IAM/SSO server (\textbf{NFR-AV-06}) matches this 99.9\% target because a single IAM failure would prevent access to all applications simultaneously. The Warehouse/WMS (\textbf{NFR-AV-03}) is set at 99.5\% measured during operational hours (07:00--18:00), as brief degraded-mode operation outside those hours is acceptable. The Accounting system (\textbf{NFR-AV-04}) and HR/Payroll system (\textbf{NFR-AV-05}) both target 99.0\%, reflecting their business-hours-only usage patterns and the fact that brief outages outside office hours do not materially affect operations.

\subsection{Performance}

Customer portal page load time must not exceed 2 seconds under normal load of 1,200 transactions per day (\textbf{NFR-PERF-01}, derived from UR-06) and must remain within 5 seconds under peak load of 10,000 transactions per day (\textbf{NFR-PERF-02}); the web cluster's auto-scaling capability must absorb the 8:1 peak-to-average ratio without manual intervention. RabbitMQ message processing latency must remain below 500~ms under normal load (\textbf{NFR-PERF-03}, derived from FR-03), and 95\% of PostgreSQL queries must complete within 200~ms under normal conditions (\textbf{NFR-PERF-04}, derived from FR-01). WAN failover from the MPLS link to the IPSec VPN must complete within 60 seconds without manual intervention (\textbf{NFR-PERF-05}).

\subsection{Security}

All data in transit between VLANs, sites, and external parties must be encrypted using TLS 1.2 as a minimum with TLS 1.3 preferred (\textbf{NFR-SEC-01}, derived from GDPR obligations). Data at rest in the HR/Payroll and Accounting clusters must be encrypted using AES-256 or equivalent (\textbf{NFR-SEC-02}, derived from UR-04 and GDPR). Multi-factor authentication must be enforced for all IT staff, management, and Finance/HR roles via the IAM (\textbf{NFR-SEC-03}), and all firewall rule changes must be logged, version-controlled, and require dual approval from IT management (\textbf{NFR-SEC-04}, derived from MR-05). Security logs must be retained for a minimum of 12 months online and 36 months in cold storage (\textbf{NFR-SEC-05}), as required by GDPR Article~5(1)(e) and incident response readiness obligations.

\subsection{Scalability and Capacity}

The web cluster must auto-scale horizontally to absorb the 8:1 peak-to-average load ratio without manual intervention (\textbf{NFR-CAP-01}). MongoDB must support horizontal shard expansion to accommodate customer base growth of up to 300\% without requiring an architectural redesign (\textbf{NFR-CAP-02}). Storage capacity for all systems must include a minimum 30\% headroom above projected 12-month utilisation (\textbf{NFR-CAP-03}). Capacity metrics must trigger a planning review alert when any critical resource sustains utilisation above 70\% of provisioned capacity for seven consecutive days (\textbf{NFR-CAP-04}).
