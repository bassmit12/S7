\section{Background and Motivation}

Enterprise infrastructure has become way more complex over the past decade. Kubernetes is now the go-to solution for running containers, but using it brings two ongoing challenges: \emph{reactive monitoring} and \emph{poor visibility into network security}.

Traditional monitoring systems rely on fixed thresholds. They're built for static, predictable workloads. But Kubernetes is anything but static — pods come and go constantly, traffic patterns change all the time, and dozens of microservices talk to each other. Fixed thresholds lead to tons of false alarms and often miss subtle issues that actually matter. Engineers get overwhelmed with alerts (alert fatigue), and problems only get noticed \emph{after} they've already affected users. It's a reactive approach, not a proactive one.

On the security side, Kubernetes networking by default uses \texttt{iptables} rules managed by \texttt{kube-proxy}. This only works at Layer~3/4 (IP addresses and ports), gives you limited insight into what's actually happening, and becomes a bottleneck when you scale up. You can't see application-level traffic details, and you have no way to detect suspicious activity like lateral movement inside your cluster.

Cilium\footnote{\url{https://cilium.io}} fixes these networking problems by using eBPF (extended Berkeley Packet Filter) — a technology that lets you run safe programs inside the Linux kernel without changing the kernel itself. Cilium can enforce network policies from Layer~3 all the way up to Layer~7 (including HTTP, gRPC, and Kafka). It encrypts traffic between workloads with mutual TLS and provides detailed, real-time network flow data through its Hubble observability component.

Here's the key insight: combining AI-driven monitoring with Cilium's network security creates something powerful. Cilium's Hubble generates detailed per-flow data that's perfect for training machine learning models to detect anomalies. At the same time, when AI detects something unusual in your infrastructure metrics, it can trigger automated responses at the network layer. Together, you get infrastructure that's \emph{self-aware} and \emph{self-defending} — it spots problems early and responds to threats automatically.

\section{Project Scope}

This project explores how to design, build, and test an integrated system that combines AI-powered monitoring with Cilium-secured Kubernetes networking to enable proactive, automated infrastructure management.

What we'll cover:
\begin{itemize}
    \item Understanding the current limitations of Kubernetes monitoring and networking
    \item Designing an integrated architecture using Prometheus, an AI anomaly detection engine, Cilium, and Hubble
    \item Building a proof-of-concept on a bare-metal Kubernetes cluster
    \item Testing how well it works and measuring the performance impact
\end{itemize}

\section{Relevance to Enterprise Infrastructure}

Infrastructure teams are dealing with growing pressure: keep services running reliably while managing increasingly complex, distributed systems. Proactive anomaly detection helps you find and fix problems faster (reducing MTTD and MTTR), which means better uptime. Meanwhile, the shift to zero-trust networking — required by frameworks like NIST SP~800-207 and many enterprise security policies — needs the kind of identity-aware, application-level network controls that Cilium provides.

By combining these capabilities, this project fills a real gap in today's enterprise tooling. The result is a practical, open-source-based approach that infrastructure teams can actually use.
